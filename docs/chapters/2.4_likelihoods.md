## Likelihoods {#sec:likelihoods} 
The top-level component `likelihoods` contains an array of likelihoods in struct format specifying mappings of distributions and observations. The corresponding distributions and observations are inserted as keys in string format referencing to distributions and observations defined in the respective top-level components, or as numbers for fixed data values. 
The combination of parameterized distributions $m_i(\theta_i)$ with observations $x_i$ generates a likelihood function 



$$
\begin{aligned} \ensuremath{\mathscr{L}}(\theta_1, \theta_2, \ldots) &=  \prod_i \text{PDF}(m_i(\theta_i), x_i) \end{aligned} 
$$



The components of a likelihood struct are: 

-   `name`: custom string 
-   `distributions`: array of strings referencing the considered     distributions 
-   `data`: array of strings referencing the used data, must be of the     same length as the array of `distributions`. Alternatively, the     data-values for single-dimensional distributions can be given     in-line. For example, this can be used for constraint terms     representing auxiliary measurements. 
-   `aux_distributions`: ([optional]{.smallcaps}) array of strings     referencing the considered auxiliary distributions defined in the     top-level component `distributions`. They can be used to encode     regularizers or penalty terms to aid the minimization. These     observed data for these distributions is implicit and not part of     `data`. 

```json title="Example: Likelihoods"
"likelihoods":[ 
  { 
    "name":"likelihood1", 
    "distributions":[ 
      "dist1", 
      "dist2", 
      "single_dimensional_dist_1", 
      ... 
    ], 
    "data":         [ 
      "data1", 
      "data2", 
      0, 
      ... 
    ], 
    "aux_distributions" : [ "regularization_term" ] 
  }, 
  ... 
] 
``` 
### Auxiliary measurements and global observables
Nuisance parameters are commonly constrained by auxiliary measurements: supplemental likelihood terms (often unit Gaussians) whose observed value (sometimes known as *global observables*) can be fixed and is generally not part of the primary data.
The mechanisms described above, can already acommodate such terms in several ways: as ordinary entry of `distributions` paired with a single-entry data set, as inline numeric value in `data`, or via `aux_distributions` (with the observed value implicit in the definition of the distribution).

For models with many constraints, listing one pair per constraint term can becom lenghty. When the constraint terms are independent and of the same kind (e.g. unit Gaussians or Poisson terms) they can instead be collected into a single pair: an `unbinned` data set with a categorical axis enumerating the constrained parameters, holding one entry per auxiliary measurement, paired with a [DiagonalGauss distribution](#sec:diag-gaussian-distribution){reference-type="ref" reference="sec:diag-gaussian-distribution"} (or [DiagonalPoisson distribution](#sec:diag-poisson-distribution){reference-type="ref" reference="sec:diag-poisson-distribution"}) that evaluates one constraint term per entry:

```json title="Example: Auxiliary measurements as a diagonal constraint"
{
  ...
  "distributions":[
    {
      "name":"constraints",
      "type":"diagonal_gauss_dist",
      "index":{"name":"np","categories":["alpha_1_in","alpha_2_in"]},
      "d_axis":"nom_np",
      "members":[
        {"category":"alpha_1_in","mu":"alpha_1"},
        {"category":"alpha_2_in","mu":"alpha_2"}
      ]
    },
    ...
  ],
  "data":[
    {
      "name":"global_observables",
      "type":"unbinned",
      "entries":[ [0,0.0], [1,0.0]],
      "axes":[
        {"name":"np","categories":["alpha_1_in","alpha_2_in"]},
        {"name":"nom_np", "min":-5, "max":5}
      ]
    },
    ...
  ],
  "likelihoods":[
    {
      "name":"likelihood1",
      "distributions":["channel1_pdf","constraints"],
      "data":["channel1_data", "global_observables"]
    }
  ]
}
```

Since every entry of an unbinned data set contributes exactly one factor to the likelihood, each constraint term enters exactly once. Encoding constraint terms this way, or as individual pairs as described above, [should]{.smallcaps} be preferred over folding them into per-channel [Product distributions](#sec:product-distributions){reference-type="ref" reference="sec:product-distribution"} as factors that do not depend on the primary observables, where the intended multiplicity of these factors in the likelihood is ambiguous.
